Project: house_monitor (package, formerly a single house-monitor.py script)
Asyncio-based Python scraper that watches Austrian real-estate sites and emails notifications about new/cheaper listings.
Production location:

Package: /opt/house-monitor/house_monitor/ (see the "Package structure" section below)
Entry point: python3 -m house_monitor.main (ExecStart in the systemd unit)
Log: tail -f /var/log/house-monitor/service.log
Service: sudo systemctl status/restart house-monitor
DB: /opt/house-monitor/seen-houses.json

Price filter: EUR_PRICE_FROM = 3000, EUR_PRICE_TO = 70000
Blacklist (no notification for these): urlaub, sommerhaus, badehütte, reserviert, stellplatz, garagen, weinkeller
SKIP_NO_PERSIST (skipped but not saved to the DB, so it still notifies if the "reserviert" status goes away): reserviert
19 active scrapers:
ImmoScout24, Dibeo, FindMyHome, Willhaben, Findheim, Wohnnet, DerStandard, Immodirekt, Raiffeisen-Immobilien, OhneMakler, ImmobIlienNet, Immokralle, Immi, Bazar, ImmobilienDe, Goldgrube, ImmoLive24, DingDong, Sonnberger
ID prefixes (in seen-houses.json): is24_, dibeo_, fmh_, wh_, fh_, wn_, ds_, imd_, ri_, om_, inet_, ik_, immi_, bazar_, imde_, gg_, il24_, dd_, sb_
Cross-platform duplicate detection: _already_seen_elsewhere() — won't send a duplicate email for the same price + a similar-enough title
Pagination patterns per scraper:

?page=N — Raiffeisen, Immodirekt, Willhaben, Wohnnet, DerStandard, OhneMakler, Findheim, ImmobIlienNet, Immokralle, Immi
&page=N (0-indexed, stateless GET) — DingDong (Drupal Views table)
/page/N/ (standard WordPress, HTTP 404 after the last page) — Sonnberger
?from=N (25/page) — (was Urbanhome, removed)
?p=N — Goldgrube
&block=N — ImmobilienDe
&entry=N — FindMyHome
REST API JSON — Bazar
__NEXT_DATA__ JSON SSR — Willhaben (fallback: HTML DOM)
/search-results/indexN.html — ImmoLive24 (session-based, see below)

Important scraper-specific knowledge:

Goldgrube: decodes raw bytes with the decode_utf8_or_latin1() helper (house_monitor/models.py) — always tries UTF-8 first, because the server sometimes mislabels a response as iso-8859-1 when its bytes are actually UTF-8 (a plain latin-1 decode would accept this without error and produce mojibake, e.g. "GrÃ¼nen" instead of "Grünen")
Price parsing: as of 2026-07-02, all 18 text-based scrapers (every scraper except Bazar) read the German-formatted price (dot=thousands separator, comma=decimal) through the shared parse_de_price(text) helper (house_monitor/models.py) — previously this same regex+replace logic was repeated verbatim in ~11 places, now it's tested in one. Bazar is the sole exception: it casts the JSON API's numeric value directly (no text price string, doesn't need the helper). ImmoScout24 and Dibeo keep their own card/text-narrowing logic (ImmoScout24: has to pick the right one out of several price-looking `<li>` elements, excluding the `/m²` one; Dibeo: has to search near the "€" sign within the card's full text, because an unscoped parse_de_price would grab the first number it finds, which could wrongly be a house number or postal code in the title) — but the final numeric normalization (dot/comma conversion, converting to float) is now delegated to the helper too; only the "which substring are we looking at" decision stayed custom. The previously documented 4 "historical leftovers" (Findheim, FindMyHome, Immi, Wohnnet) have been migrated — each was tested individually against edge cases before migrating (e.g. FindMyHome's "30.000,-" trailing-dash format), the 68 tests stayed green afterward.
Willhaben: reads from the __NEXT_DATA__ JSON, filters out per-m² prices (excludes if price × area > EUR_PRICE_TO)
Findheim: the server-side price filter is unreliable → client-side safety filter in the main loop. 2026-07-01: the site got a Tailwind redesign, the old card selector (div.group + "rounded-md") and price container (div.z-[1]/right-0/bottom-0) stopped working, which caused it to silently return 0 results for MONTHS (even though real, in-range listings existed). Fixed: card = div.group + "overflow-hidden" + "border" in the class list; price = the `<p class="font-semibold ...">€ X</p>` next to the title. Lesson: a scraper returning "0 listings" does NOT automatically mean there's no inventory — it has to be checked live (without the card-count filter, then with price parsing), because a site redesign can just as easily silently break the selectors.
Immokralle: uses urllib instead of aiohttp (because of how aiohttp percent-encodes the `[ ]` characters in the URL)
Urbanhome.at: removed — ASP.NET session-based, can't be scraped automatically
Kurier (immo.kurier.at): removed on 2026-07-01 — the site's real-estate section is gone, every /suche search redirects with HTTP 302 to dibeo.at/obj/miete/v (RENTALS, not sales) (utm_campaign=immokurier_facade), regardless of the region/type parameters. The 27-URL (9 regions × 3 types) scraper was structurally dead, could never produce a real result, it was just needlessly slowing down the run.
ImmoLive24: a Flynax-based portal, Category_ID=228 = Houses/Buy. Page 1 has to be fetched with POST (post_form_key=immobilien_quick, f[kaufpreis][from]/[to]), which sets the filter in server-side session state keyed by PHPSESSID; subsequent pages come via plain GET (/immobilien/search-results/indexN.html), the shared aiohttp.ClientSession's cookie jar keeps the session. After the last page: 0 cards, HTTP 200 (no explicit "no more pages" signal). The title (h2 > a, "Haus, PLZ, Ort") is less informative than the descriptive text inside the card (the first div of td.fields) — that's used as the title, the address is only a fallback.
DingDong: a Drupal Views table (table.views-table), a partner of immokralle.com (linked in the footer), but cross-platform duplicate detection (_already_seen_elsewhere) handles any overlap. No numeric listing ID in the link, just a unique slug ("/immobilien/<slug>") — that slug is the identifier (dd_<slug>). Stateless pagination: &page=N (0-indexed), every page comes with the full filter query string, no session/cookie needed (unlike ImmoLive24).
Sonnberger: WordPress + Houzez theme, no server-side price filter parameter (only ?sortby=a_price ascending sort) — fully client-side filtering, like Findheim. For "Reserviert"/"Verkauft"/"auf Anfrage" status listings, this text appears in place of the price in list view with no digits, so parse_de_price naturally returns 0.0 for them, which the existing price==0 client filter already excludes anyway — no separate reserved/sold handling needed. ID: the [data-listid] attribute inside the card (sb_<id>). Pagination: /page/N/?sortby=a_price, HTTP 404 after the last page (not 0 cards).

Email: Gmail SMTP — EMAIL_CONFIG now reads from env vars (SMTP_SERVER/SMTP_PORT/SENDER_EMAIL/SENDER_PASSWORD/RECIPIENT_EMAIL — unprefixed since 2026-07-13; the old `HOUSE_MONITOR_` prefix was dropped so both this project and product-monitor can share the same keys), no hardcoded credentials in the code. The actual values live in a shared `/opt/secrets.env` file (used by both the house-monitor and product-monitor projects for the same Gmail account — a systemd `EnvironmentFile=/opt/secrets.env` loads it in production). The `.get()` fallback (empty string) means a missing env var doesn't raise at import/test-collection time either. The legacy `PRODUCT_MONITOR_*` keys are still present in that shared file for now, but as of 2026-07-03 `product-monitor.service`'s systemd unit has no `EnvironmentFile=` line yet (it still reads credentials from `product-monitor-config.json` instead) — when product-monitor migrates to env-var-driven config, it should read the shared unprefixed keys, add `EnvironmentFile=/opt/secrets.env` to its unit, and the `PRODUCT_MONITOR_*` lines can then be deleted from secrets.env.
Email body design: as of 2026-07-06, _build_email_body() is a single flat, price-ordered list — no per-source grouping/header anymore (it used to be modeled on the sibling project's, product-monitor.py's, per-source "==== source (N listings) ====" blocks, one such block per source, sources in alphabetical order; that grouping was dropped). Ordering: new listings first (ascending by price), then price changes (also ascending by price) — across ALL sources combined, not per-source. Each entry shows the source name inline in parentheses right after the title (e.g. "Cozy house (Sonnberger)"), using the short display name from the SOURCE_DISPLAY_NAMES dict — Listing.source's raw value (e.g. "sonnberger.co.at") is never shown. A "-" * 148 separator appears before the first entry and after every entry (there's no more per-source block boundary, so it's just per-listing now). No emoji in the text. The price's thousands separator is a space (the _fmt_count helper, same as the console summaries). Subject: "Ingatlanok: {N} db" (Hungarian: "Real estate: {N} listings" — the email itself is in Hungarian, the owner's language).
Scheduling: live since 2026-07-02 — _seconds_until_1600() + asyncio.sleep() at the top of the while True loop, the process itself never exits (no break). The systemd unit's `Restart=always` + `RestartSec=300` + `StartLimitIntervalSec=0` (no restart limit) makes sure it restarts even after an unexpected crash; since _seconds_until_1600() recalculates on every startup, if the failure happened after 16:00 that day, the next attempt automatically slides to the next day.
Lock: /tmp/house-monitor.lock — prevents overlapping runs.
HTTP timeout: a session-level default of 30s (aiohttp.ClientTimeout when the ClientSession is created in run()) — previously only 4 scrapers set an explicit timeout, the rest relied on aiohttp's 300s default

**Parallelizing scraper runs:**
- The 19 scrapers run concurrently in run() via `asyncio.gather(*(s.fetch_listings() for s in scrapers), return_exceptions=True)` (previously they ran strictly sequentially, one after another, which needlessly slowed down the whole run even though the scrapers hit independent domains and share no state besides the common aiohttp.ClientSession)
- Because of `return_exceptions=True`, a scraper raising an exception doesn't sink the whole run — only that one source is skipped (and logged), the rest continue undisturbed
- For scrapers that crawl multiple independent base URLs (OhneMakler: 2, Immokralle: 2, ImmobilienDe: 4 category URLs), fetch_listings() is split into a `_fetch_one_url(base_url)` helper method, which is also called per-URL concurrently via `asyncio.gather`; the results are then merged afterward with a single shared `seen_ids` set, so any overlap (duplicate ID) between base URLs doesn't end up twice in the final result

**Connection-drop retry + same-day retry of failed sources (2026-07-13):**
- Root cause of the recurring `[Errno 32] Broken pipe` (daily on Findheim page 2, occasionally Raiffeisen/Willhaben): the shared session's keep-alive pool hands out a socket the server already closed while a slow page was being processed — the next pagination request writes into the dead connection. On 2026-07-12 it hit Findheim's page 1 → "Findheim: 0 listings" for that run (selectors were fine, verified live).
- First line of defense: `fetch_text(session, url)` (house_monitor/fetch.py) — GET returning `(status, body)`, retries `aiohttp.ClientConnectionError` (covers ClientOSError/broken pipe AND ServerDisconnectedError) up to 3 attempts with 2s/4s backoff; the retry runs on a fresh connection (aiohttp evicts the broken socket), so one retry almost always succeeds. Deliberately NOT retried: non-200 statuses (callers' own status handling stays in charge) and timeouts (retrying those at the 16:00 concurrent peak would only slow the run). Currently used by the 4 scrapers with observed failures (Findheim, Raiffeisen, Willhaben; Goldgrube since 2026-07-13 after a live "Server disconnected" hit — it uses the `fetch_bytes` sibling helper, which returns (status, raw bytes, declared charset) untouched so `decode_utf8_or_latin1` stays in charge of decoding; plain fetch_text would re-introduce the mojibake bug) — migrate others individually if they start failing too, same as the parse_de_price migration pattern.
- Second line: same-day source-level retry. Each of the 4 scrapers sets `self.incomplete = True` when fetch_listings() ends on an error (partial results are still returned and processed). The old run() body is now `_scrape_and_notify(scrapers) -> failed_scrapers` (returns gather-level crashers + `incomplete`-flagged ones, via `getattr(scraper, "incomplete", False)` — scrapers without the flag work unchanged). The main loop then retries ONLY the failed sources after `INCOMPLETE_RETRY_DELAYS = (1800, 1800, 3600)` (config.py) — the main 16:00 email is never delayed; anything new found on retry goes out as a separate follow-up email the same day; the seen-DB + cross-platform dup filter prevent double-notification.

**Logging — the NOTICE level (compressing the console for manual runs):**
- `NOTICE = 25` is a custom log level (between INFO=20 and WARNING=30), used via the `log_notice()` helper
- On a manual run (not systemd), the console handler only prints NOTICE+ (milestones: run start/end, DB loaded, total result count, per-scraper summary, email outcome); the detailed per-page/per-listing INFO logs only go to the file (house-monitor.log)
- Console format on a manual run: `%H:%M message` (e.g. `19:12 Searching...`) — no date, seconds, or level name; the file/systemd log format is unchanged (full date+time+level, for debugging)
- The "Logging initialized..." line now goes through plain `logging.info()` (not `log_notice()`), so it doesn't show up on the console, only in the file
- Per-scraper summary lines in a compact format: `{Source}: {N} listings` or `{Source}: {N} ads` (the source name without the domain suffix, e.g. "FindMyHome" not "FindMyHome.at" — except where the TLD is the distinguishing part: Immobilien.net vs Immobilien.de)
- Unchanged in systemd mode (INVOCATION_ID env var): there's a single stdout stream, every INFO detail stays in it — it ends up in service.log, which CLAUDE.md documents as the place to debug from, so full detail is kept there

**DB reliability (seen-houses.json):**
- _load_db(): loads entries line by line, a single corrupt/incompatible entry is just skipped (and logged), it doesn't throw out the whole DB — previously a single bad entry would have taken down the entire thing, which would have caused a flood of duplicate-email spam (every already-seen house would have looked "new")
- _save_db(): atomic write — writes to `seen-houses.json.tmp` first, then renames it to the final name with `os.replace()`, so an interrupted write (power loss, kill) can't corrupt or empty out the existing database

**Package structure (since the 2026-07-01 modularization):**
```
house_monitor/
├── __init__.py          — minimal, no re-exports (see below for why)
├── models.py            — Listing dataclass, decode_utf8_or_latin1, parse_de_price
├── fetch.py             — fetch_text/fetch_bytes: GET with retry on connection-level errors (see below)
├── config.py            — all constants (BLACKLIST, SKIP_NO_PERSIST, EUR_PRICE_*, EMAIL_CONFIG, per-scraper URLs)
├── logging_setup.py      — NOTICE, log_notice, _fmt_count
├── email_notifier.py     — EmailNotifier
├── scrapers/
│   ├── __init__.py        — re-exports all 19 scraper classes + __all__
│   └── <one file per scraper>  (e.g. sonnberger.py, immolive24.py, ...)
├── monitor.py            — HouseMonitor, SCRAPER_SUMMARY_LABELS, SOURCE_DISPLAY_NAMES
└── main.py               — the old `if __name__ == "__main__":` lock+run block, as a main() function
```
- The tests' (`tests/test_house_monitor.py`) `hm` fixture aliases the `house_monitor.monitor` module directly (NOT the package's `__init__.py`!) — this matters because some tests directly modify `hm.DATA_FILE` and monkeypatch `hm.os.replace`, then call the real `_save_db`/`_load_db` methods, which read those same names from that same module namespace. If `hm` were the package `__init__.py` with a re-export, the monkeypatch would only modify a copy, and the actual code wouldn't notice — that's also why `house_monitor/__init__.py` stays deliberately empty (just `__version__`), with no re-exports.
- `tests/conftest.py` (3 lines) sets up `sys.path` so `from house_monitor import monitor` works without installing the package (`pip install -e .`) — the Pi has no internet/pip either, the package is just deployed by copying files (rsync/sshfs), same as the single .py file used to be.

**Development pattern:**
- The scraper list and the BLACKLIST keep growing — new sources and filter words get added regularly
- When adding a new scraper, always: constants into `config.py` → the class into `scrapers/<new_file>.py` → the import+`__all__` into `scrapers/__init__.py` → the `scrapers` list and `SCRAPER_SUMMARY_LABELS`/`SOURCE_DISPLAY_NAMES` in `monitor.py` → any client-side price filter needed
- Encoding, pagination, and duplicate detection each need individual attention for every new scraper

**Server:**
- Raspberry Pi (ARM Linux, Ubuntu server based)
- Python 3.11+, systemd service
- Limited RAM → avoid high memory usage (e.g. don't load all pages at once)

**Tests:**
- tests/test_house_monitor.py — pytest, no network calls/email sends, just parsing/filtering/serialization logic
- Run: `python3 -m pytest tests/ -v` from the project root (`tests/conftest.py` sets up sys.path so the `house_monitor` package is importable)
- Dependencies: `pip install -r requirements.txt` (aiohttp, aiofiles, beautifulsoup4, lxml, pytest)
- 2026-07-03: verified the full suite (68/68) also passes in a genuinely clean venv with nothing but the declared requirements.txt installed - not just the long-lived dev environment, which can accidentally hide a missing/undeclared dependency. Recipe for a system Python without `ensurepip` (`python3 -m venv` fails with "ensurepip is not available"): `python3 -m venv --without-pip .venv`, then bootstrap pip via `curl -sS https://bootstrap.pypa.io/get-pip.py -o get-pip.py && .venv/bin/python3 get-pip.py`, then `.venv/bin/pip install -r requirements.txt`.
- Covers: decode_utf8_or_latin1 (Goldgrube encoding), parse_de_price (shared price-parsing helper), Goldgrube price parsing, _titles_similar / _already_seen_elsewhere (duplicate detection), _build_email_body, seen-houses.json save/load round-trip, _load_db's fault tolerance (a single corrupt entry doesn't zero out the DB), _save_db's atomicity (no lost/truncated data on an interrupted write), fetch_text/fetch_bytes retry behavior (retries broken pipe, re-raises after 3 attempts, doesn't retry non-200, fetch_bytes hands the raw body + declared charset through untouched), _scrape_and_notify's failed-source collection (crashed + incomplete scrapers queued for retry, partial results still notified), a dedicated card/JSON-parsing test for every one of the 19 scrapers, every scraper's instantiability — 74 tests total
- When adding a new scraper/logic, it's worth adding a test here too, so future changes don't silently break anything
- 2026-07-02: every one of the 19 scrapers' card-parsing logic was pulled out into a standalone, network-free-testable `_parse_cards(html_text)` method (`_parse_items`/`_parse_json_adverts` for the JSON-based scrapers) — previously only 5 scrapers (ImmobIlienNet, Immokralle, ImmoLive24, DingDong, Sonnberger) had one, the rest had their card processing embedded directly in the paginating `fetch_listings()`/`_fetch_one_url()` loop. `_parse_cards` only does the raw extraction (a list of id/title/url/price tuples); dedup (`seen_ids`) and price-range filtering stay on the caller's side (fetch_listings) — this pattern was already the precedent set by the 5 pre-existing scrapers.
- IMPORTANT LIMITATION: the tests run against hand-built, static HTML/JSON fixtures, with no real network calls — this means the test suite CANNOT by itself catch a live site redesign (like Findheim's 2026-07-01 case above), it only prevents a regression of an already-known, already-fixed bug. A live check (card count/price parsing via a manual run) is needed if a source suspiciously returns 0 results.
- `scripts/live_smoke_check.py` (2026-07-02): a live-check script written in response to the limitation above — `python3 -m scripts.live_smoke_check` runs all 19 scrapers against the live sites, with no price/blacklist filtering and no email/DB writes, printing the result count per source (flagging 0-result/errored sources). Deliberately NOT part of pytest (network-dependent, slow, non-deterministic) — a manual/periodic diagnostic tool. The session has to send the SAME browser headers (User-Agent, Accept, Accept-Language) as the real `monitor.py` — without them, several sites (ImmoScout24, Immodirekt, Immobilien.net — all the same ImmoScout24 AT portal) block with HTTP 401, which would give a false "broken" signal.
- 2026-07-02: the first run flagged 6 sources as suspicious; on investigation, all 6 turned out to have no real bug. The 3 HTTP 401s (ImmoScout24, Immodirekt, Immobilien.net) were the script's own fault (missing headers, see above), not a site issue. The other 3 (OhneMakler, Immobilien.de, ImmoLive24) returned HTTP 200 with 0 cards — confirmed via manual curl with a raised price ceiling (100k/150k/300k) that the selectors/search parameters work correctly, there just genuinely was no house between 3000-70000 € on any of those sites at that moment (OhneMakler: explicitly empty GeoJSON; Immobilien.de: a "keine Treffer gefunden" ["no results found"] message on the page; ImmoLive24: the POST-based price filter scales correctly, it just genuinely returns 0 under 70k). Lesson: "0 results" alone really isn't enough to declare something broken — always confirm with a raised price ceiling/wider filter before calling a scraper "broken."
- Dev tooling (formatting/lint/type-checking, NOT needed to run the scraper itself): `requirements-dev.txt` (black, ruff, mypy, types-beautifulsoup4), configured in `pyproject.toml`. The `[tool.mypy.overrides]` section deliberately disables a few error codes (`union-attr`, `index`, `operator`, `call-arg`, `arg-type`, `assignment`) for the `house_monitor.scrapers.*` modules — this is noise from BeautifulSoup's imprecise types (e.g. `str | list[str] | None`), not a real bug; the root `house_monitor/` modules (monitor.py, email_notifier.py, etc.) run under full strictness.
- Dependabot (`.github/dependabot.yml`, added 2026-07-02) opens weekly PRs for outdated pip packages and GitHub Actions versions. A green CI run on a Dependabot PR only proves the network-free unit tests still pass — it does NOT prove the real scrapers still work, since aiohttp/bs4/lxml version bumps can change real network/parsing behavior that no unit test exercises. 2026-07-02 incident: the `aiohttp` 3.9.1 → 3.14.1 PR had green CI, but running `scripts/live_smoke_check.py` against it repeatedly produced `ClientPayloadError` on a different subset of scrapers each run (Findheim, Wohnnet, Raiffeisen, ImmoLive24, Immobilien.de) — a concurrency-related regression under this project's `asyncio.gather`-on-one-shared-session usage pattern. Reverting to aiohttp==3.9.1 reproduced zero errors across repeated runs. Rule going forward: any aiohttp/bs4/lxml Dependabot PR needs a `scripts/live_smoke_check.py` run against the live sites (ideally 2+ times, since this failure mode is intermittent) before merging, not just a green CI check.
